# jiraACP — Product Specification

> AI-powered project pipeline CLI: Jira → Code → GitHub → Deploy → Notify

## 1. Overview

jiraACP is a project-centric CLI that automates the full ticket lifecycle. Each **project session** binds one Jira instance + one GitHub repo + one Claude Code workspace + one Telegram channel into a single orchestrated unit.

**Core value**: A developer runs `jiraACP run` in the morning. By the time they check Telegram, tickets have been analyzed, coded, reviewed, deployed, and tested. Ambiguous tickets prompt a clarification message instead of silently failing.

### Installation

```bash
npm install -g jira-acp

jiraACP init      # interactive setup wizard
jiraACP run       # start pipeline for assigned tickets
```

### Two binaries, one package

| Binary | Purpose |
|--------|---------|
| `jiraACP` | Pipeline CLI — all commands |
| `jiraACP-mcp` | Bundled Jira MCP server — auto-configured for Claude Code agents |

`jiraACP init` writes `.mcp.json` into the workspace so Claude Code agents have Jira tools available without any manual config.

---

## 2. Problem Statement

Small dev teams using Jira + GitHub lose significant time on repetitive ceremony:
- Manually reading tickets, creating branches, writing PR descriptions
- Context-switching between Jira, GitHub, terminal, and Telegram
- Chasing unclear requirements before coding can start
- Updating Jira status and notifying teammates after completion

jiraACP automates the mechanical parts while keeping humans in the loop for decisions that matter.

---

## 3. Key Features

### 3.1 Project Sessions
Each project has its own config: Jira credentials, GitHub repo, Claude workspace, Telegram chat, deploy scripts. Multiple projects run independently.

### 3.2 Automated 9-Stage Pipeline

| Stage | What happens | Agent |
|-------|-------------|-------|
| 1. Fetch | Get assigned Jira tickets | Haiku |
| 2. Analyze | Clarity score: criteria, design, dependencies | Sonnet |
| 3. Clarify | Send Telegram if ambiguous, wait for human | Haiku |
| 4. Code | Claude Code implements the ticket | Sonnet/Opus |
| 5. Git | Create branch, commit, push, open PR | Haiku |
| 6. Review | Review agent checks PR, auto-merge if clean | Sonnet |
| 7. Deploy | Run deploy script to dev server | Haiku |
| 8. Test | Playwright agent tests on dev server | Sonnet |
| 9. Notify | Jira: comment + transition + reassign. Telegram: done | Haiku |

### 3.3 Human-in-the-Loop via Telegram
- Ambiguous tickets → Telegram clarification with structured reply format
- Review failures → Approve/Reject buttons
- Configurable timeouts with skip/abort/proceed actions

### 3.4 Self-Contained Jira Integration
- Bundles the Jira MCP server (`jiraACP-mcp` binary)
- Auto-generates `.mcp.json` so Claude agents have Jira tools
- Multi-tenant: each project maps to one Jira instance via env vars

### 3.5 Auto-Generated Claude Memory
- `CLAUDE.md`: project architecture, stack, conventions (committed to repo)
- Per-run memory files: ticket context, coding plan, review feedback
- Generated by `jiraACP init` and refreshed by `jiraACP update-context`

### 3.6 Crash-Safe State
- Append-only event log per ticket run
- Resume from any stage after crash: `jiraACP resume PROJ-123`
- Pending Telegram clarifications survive bot restarts

### 3.7 Additional Features
- `--dry-run`: full simulation, no side effects
- `--from/--to`: run a subset of stages
- `jiraACP sprint`: parallel pipelines for all sprint tickets
- `jiraACP triage`: clarity analysis only, no code
- `jiraACP schedule`: cron-based scheduled runs
- `jiraACP serve`: webhook trigger for Jira automation rules
- `jiraACP dashboard`: terminal UI of active runs
- `jiraACP usage`: token cost tracking per project/month

## 4. CLI Commands

### Setup
```
jiraACP init [--dir <path>]           Interactive setup wizard
jiraACP doctor [--fix]                Health-check all integrations
jiraACP projects [list|add|remove]    Manage known projects
jiraACP update-context                Regenerate CLAUDE.md from codebase
```

### Pipeline
```
jiraACP run <ticketKey> [options]     Run pipeline for one ticket
  --project <name>                    Target project (default: cwd auto-detect)
  --from <stage>                      Start from stage
  --to <stage>                        End at stage
  --dry-run                           Simulate, no side effects
  --no-confirm                        Non-interactive mode

jiraACP sprint [options]              Run all sprint tickets
  --project, --sprint, --parallel, --filter, --dry-run

jiraACP triage [options]              Clarity analysis only
```

### Monitoring
```
jiraACP status [ticketKey]            Active pipeline states
jiraACP dashboard [--watch]           Terminal UI
jiraACP logs <ticketKey> [--stage] [--follow]
jiraACP replay <ticketKey>            Replay completed run
jiraACP usage [--month YYYY-MM]       Token cost report
```

### Control
```
jiraACP pause <ticketKey>
jiraACP resume <ticketKey>
jiraACP abort <ticketKey> [--reason]
jiraACP cancel-clarification <ticketKey>
```

### Schedule
```
jiraACP schedule add --cron "0 9 * * 1-5" --project <name>
jiraACP schedule list
jiraACP schedule remove <id>
```

### Config
```
jiraACP config get [<key>]
jiraACP config set <key> <value>
jiraACP config edit
```

### Utility
```
jiraACP serve [--port 3100]           Webhook server for external triggers
```

### Exit Codes
| Code | Meaning |
|------|---------|
| 0 | Success |
| 1 | Pipeline stage failed |
| 2 | Configuration error |
| 3 | Integration unreachable (Jira/GitHub/Telegram) |
| 4 | User aborted |
| 5 | Already running (lock conflict) |

## 5. Project Config Schema

Config lives at `{workspace}/.jira-acp/project.json`. Sensitive values use `"env:VAR_NAME"` references — safe to commit.

```typescript
{
  name: string                          // "my-saas"
  extends?: string                      // path to base config

  jira: {
    instance: string                    // matches JIRA_{INSTANCE}_* env vars
    projectKey: string                  // "PROJ"
    assignees: string[]                 // Jira accountIds
    acceptanceCriteriaField: string     // default: "customfield_10016"
    inProgressTransition: string        // default: "In Progress"
    inReviewTransition: string          // default: "In Review"
    doneTransition: string             // default: "Done"
    clarityScoreThreshold: number       // 0-1, default: 0.7
    requiredFields: string[]            // ["description", "acceptanceCriteria"]
  }

  github: {
    owner: string
    repo: string
    defaultBranch: string               // default: "main"
    branchPattern: string               // "{prefix}/{ticketKey}-{slug}"
    autoMergeStrategy: "squash"|"merge"|"rebase"
    reviewers: string[]                 // GitHub usernames
    majorIssueThreshold: number         // default: 1
  }

  workspace: {
    rootDir: string                     // absolute path to codebase
    buildCommand?: string               // "npm run build"
    allowedPaths: string[]              // paths agent can modify
  }

  deploy: {
    enabled: boolean
    command?: string                    // "npm run deploy:dev"
    rollbackCommand?: string
    timeoutMs: number                   // default: 1_200_000
    healthCheckUrl?: string
  }

  test: {
    enabled: boolean
    baseUrl?: string
    retries: number                     // default: 2
    waitBeforeTestMs: number            // default: 5_000
    specPattern: string                 // "e2e/**/*.spec.ts"
  }

  telegram: {
    botToken: string                    // "env:TELEGRAM_BOT_TOKEN"
    chatId: string | number
    notifyOnStages: Stage[]
    humanInTheLoop: {
      clarificationTimeoutMs: number    // default: 3_600_000 (1hr)
      clarificationTimeoutAction: "skip"|"abort"|"proceed-with-warning"
      reviewApprovalTimeoutMs: number   // default: 86_400_000 (24hr)
      reviewApprovalTimeoutAction: "abort"|"merge-anyway"
    }
  }

  pipeline: {
    maxConcurrentRuns: number           // default: 2
    stageTimeouts: Record<Stage, number>
    agentStallTimeoutMs: number         // default: 300_000
    skipClarificationIfClear: boolean   // default: true
    failOnTestFailure: boolean          // default: false
    maxCostUsdPerRun?: number
    hooks: {
      beforePipeline?: string
      beforeCode?: string
      afterCode?: string
      afterDeploy?: string
      afterPipeline?: string
    }
  }
}
```

## 6. Human-in-the-Loop Rules

### Always ask when:
| Trigger | What to send |
|---------|-------------|
| Clarity score < threshold | Numbered missing items + `/answer PROJ-123` template |
| Review agent: major issues ≥ threshold | PR diff annotations + Approve/Reject buttons |
| Merge conflict | Conflicting files list + "I'll resolve" button |
| Deploy target unreachable | "Retry / Skip deploy" buttons |
| Tests fail after 3 retries | Playwright screenshot + error + Re-run/Close buttons |
| Cost about to exceed `maxCostUsdPerRun` | "Continue / Abort?" |

### Never ask when:
- Clarity score ≥ threshold and all required fields present
- PR review: minor issues only (style, typos)
- Build passes first try
- All tests pass

### Timeout flow (clarification, 1hr default):
```
T+0    → Send Telegram with questions
T+30m  → Send reminder
T+45m  → "Pipeline skips in 15min"
T+60m  → Execute clarificationTimeoutAction
          skip: comment ticket, move to next
          abort: kill pipeline, notify
          proceed-with-warning: continue, log ambiguities in Jira comment
```

### Telegram reply format:
```
[jiraACP] PROJ-123: Needs Clarification

Missing:
1. Authentication method? (ticket says "use auth", no method specified)
2. Mobile responsive?

Reply with:
/answer PROJ-123
1. JWT, same as existing endpoints
2. Yes, mobile-first

[Skip this ticket]  [Abort pipeline]
```

## 7. Agent Memory Architecture

### CLAUDE.md (stable, committed to repo)
Generated by `jiraACP init`, updated by `jiraACP update-context`. Contains:
- Architecture overview
- Tech stack
- Coding standards
- Critical files (do not modify without review)
- Available MCP tools
- Workflow rules (branch naming, commit format, forbidden actions)

### Per-run memory files (`{stateDir}/runs/{ticketKey}/memory/`)
| File | Created at | Used by |
|------|-----------|---------|
| `ticket-context.md` | Stage 1 | Stages 2-9 |
| `code-plan.md` | Stage 4 start | Stage 4 (checkpointing) |
| `review-feedback.md` | Stage 6 | Stages 6-9 |
| `run-metadata.json` | Stage 1 | State machine |

### .mcp.json (generated by `jiraACP init`)
```json
{
  "mcpServers": {
    "jira": {
      "command": "jiraACP-mcp",
      "env": { "JIRA_MYPROJECT_URL": "env:JIRA_MYPROJECT_URL", ... }
    }
  }
}
```

## 8. State Persistence

### Directory structure
```
{workspace}/.jira-acp/
├── project.json
├── runs/
│   └── PROJ-123/
│       ├── state.json          # Append-only event log
│       ├── PROJ-123.lock       # PID + start time
│       ├── memory/             # Per-run context files
│       └── logs/               # Per-stage agent output
└── pending-clarifications.json # Active Telegram questions
```

### Event log (append-only, crash-safe)
```typescript
type PipelineEvent =
  | { type: "STARTED"; timestamp: string }
  | { type: "STAGE_STARTED"; stage: StageId; timestamp: string }
  | { type: "STAGE_COMPLETED"; stage: StageId; output: unknown; timestamp: string }
  | { type: "STAGE_FAILED"; stage: StageId; error: string; timestamp: string }
  | { type: "CLARIFICATION_REQUESTED"; questions: string[]; timestamp: string }
  | { type: "CLARIFICATION_RECEIVED"; answers: string; timestamp: string }
  | { type: "PIPELINE_COMPLETED" | "PIPELINE_ABORTED"; timestamp: string }
```

Resume logic: replay event log → compute current stage → restart from last incomplete stage.

## 9. Security

- No plaintext secrets in `project.json` (use `"env:VAR_NAME"` references)
- `jiraACP doctor` warns if secrets detected in git-tracked config
- Claude agent subprocess: minimal env whitelist (`PATH`, `HOME`, `ANTHROPIC_API_KEY`, project vars)
- Agent writes scoped to `workspace.allowedPaths`
- Telegram: ignore messages from non-configured chatIds
- GitHub token minimum scopes: `repo`, `contents:write`, `pull_requests:write`, `actions:read`
- Lock files via `O_EXCL` (atomic, crash-safe)
- All external state mutations logged to `audit.log` with tokens redacted

## 10. Non-Goals (v1)

1. Multi-Jira-instance per project
2. Staging/production deploys (dev only — safety boundary)
3. Database migration automation
4. Slack/Discord notifications (Telegram only)
5. LLM provider switching (Claude Code only)
6. Ticket creation (jiraACP consumes tickets, never creates)
7. Windows support
8. Self-hosted LLM
9. Real-time multi-human collaboration on same pipeline run
10. Full code review replacement (humans still own GitHub reviews)

---

## 11. Success Metrics

- Time from ticket assignment to PR open: < 30min for well-defined tickets
- Clarification rate: < 20% of tickets need human input
- Pipeline resume success: 100% of crashed runs resumable
- Zero plaintext secrets in any generated file
