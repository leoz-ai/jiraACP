## 2. Sprint 2 Tasks — Architecture Fit Assessment

### TASK-001 — `abort` command

**Assessment: SAFE**

Lock file stores `{ pid, startedAt, stage }`. Abort reads lock → `process.kill(pid, "SIGTERM")`. The target process already handles SIGTERM in `acquireLock` (cleanup + exit 143). Abort command then emits `PIPELINE_ABORTED` to the state file independently.

**Risk**: Race condition if lock is released between read and kill. Mitigation: wrap in try/catch, treat ESRCH as "already dead — still emit abort event".

**New file needed**: `src/commands/abort.ts`

---

### TASK-002 — `triage` command

**Assessment: SAFE — reuse existing stages**

Stage 1 (`fetchStage`) and Stage 2 (`analyzeStage`) are pure functions that take a `PipelineContext`. Triage creates a minimal context per ticket (no lock, no state file) and runs them. Parallel dispatch via `Promise.allSettled` (max 5 concurrent).

**Risk**: `analyzeStage` calls `runAgent` which spawns Claude subprocess — 5 concurrent spawns is fine for Node 20.

**No new infrastructure needed** — reuse `getClient`, `createLogger`, `analyzeStage.run()`.

### TASK-003 — `schedule` command

**Assessment: SAFE — JSON storage only**

No cron runtime dep needed for storage. Schedule entries stored at `~/.jira-acp/schedules.json`:
```typescript
interface ScheduleEntry {
  id: string        // uuid v4
  projectName: string
  cron: string      // validated as cron expression
  createdAt: string
  enabled: boolean
}
```
Cron validation: basic regex `^(\S+\s){4}\S+$` is sufficient for v1. Execution runtime lives in `serve` command (TASK-005).

**New file needed**: `src/commands/schedule.ts`
**No new npm dep** — avoid adding `node-cron` package; use `setInterval` with manual cron parsing for v1, or validate cron string and document that `serve` executes it.

**Recommendation**: For v1, use a simple polling approach in `serve` (check schedules every minute, compare current time to cron expression). Add `cron` npm package only if needed. Document in TASKS.md.

### TASK-004 — `dashboard` command

**Assessment: SAFE — read-only state aggregation**

Reads `~/.jira-acp/runs/<project>/<ticket>/state.json` across all projects. Compute derived state via existing `StateManager`. Render with `picocolors` (already in deps). `--watch` uses `setInterval(render, 5000)` + ANSI clear.

**Risk**: Large number of old runs could slow down. Mitigation: only show runs from last 24h by default.

**New file needed**: `src/commands/dashboard.ts`

### TASK-005 — `serve` command

**Assessment: MODERATE — new HTTP server, no new dep**

Use Node.js `node:http` only. No express/fastify dep needed.

```
POST /webhook/jira  → validate X-Jira-ACP-Secret → runPipeline async
GET  /health        → { status, version, uptime }
```

Cron schedule execution: load `~/.jira-acp/schedules.json` → parse each cron → compute next fire time → `setTimeout` to next occurrence → loop.

**Risk**: Long-running `runPipeline` calls block nothing (they're async + subprocess). Safe.

**New file needed**: `src/commands/serve.ts`

### TASK-006 — `sprint` command

**Assessment: MODERATE — DAG + concurrency**

Fetch sprint tickets via Jira client (existing `getClient`). Build dependency graph from `issueLinks` (`blocks`/`is blocked by`). Topological sort → schedule independent tickets in parallel batches of `maxConcurrentRuns`.

`Promise.allSettled` for each batch → collect results → log summary.

Ctrl+C during sprint: install `SIGINT` handler → `abort` all running tickets (TASK-001 dependency justified).

**Risk**: Jira `issueLinks` field naming varies by instance. Mitigation: check both `blocks` and `is-blocking` link types.

**New file needed**: `src/commands/sprint.ts`

### TASK-007 — Telegram approval flow (Stage 6 hardening)

**Assessment: SAFE — extends existing human-loop pattern**

`requestClarification` in `human-loop.ts` already has the Promise + pending store pattern. `requestApproval` follows identical pattern but uses `callback_query` instead of `/answer` command.

grammY `bot.on("callback_query:data", handler)` — already supported by grammY v1.30.

State events `HUMAN_APPROVAL_REQUESTED` / `HUMAN_APPROVED` / `HUMAN_REJECTED` already defined in `state.ts`.

Stage 6 change: replace `throw new Error("REVIEW_FAILED")` with `await requestApproval(...)`. If rejected → throw `REVIEW_REJECTED` error. If approved → proceed to merge.

**Risk**: Inline keyboard buttons are one-time use; handle duplicate callbacks gracefully (already-resolved promise → ignore).
